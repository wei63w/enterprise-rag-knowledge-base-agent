# Enterprise RAG Knowledge Base Agent - MVP 范围、验收标准与异常处理

## 1. 文档目的

本文补充主设计文档的实施边界，明确第一版 MVP 要做什么、做到什么程度算完成，以及常见失败场景如何处理。

MVP 的目标不是一次性实现完整企业级平台，而是交付一条可运行、可演示、可验证的 RAG 主链路：上传文档、解析切片、建立索引、发起问答、返回答案并展示引用来源。

## 2. MVP 范围

### 2.1 本期包含

**文档处理**

- 支持上传 PDF、Markdown、TXT，Word `.docx` 可作为候选格式，若实现成本过高可延后到下一阶段。
- 上传后的原始文件保存到 MinIO。
- 文档元数据保存到 MySQL，包括文件名、格式、大小、处理状态、错误信息、切片数量和时间戳。
- 使用 Apache Tika 完成基础文本提取。
- 默认使用固定长度或滑动窗口切片策略，优先保证实现稳定和结果可解释。
- 对切片内容生成 embedding，并写入 Milvus。

**检索问答**

- 支持用户在前端输入问题并创建一次问答请求。
- 后端根据问题执行向量检索，取 Top-K 相关切片构造上下文。
- 调用默认 LLM 生成答案。
- 当检索结果为空或置信度不足时，返回“未在知识库中找到足够依据”的兜底回答。
- 问答记录保存到 MySQL，短期会话上下文可使用 Redis 保存最近若干轮。

**答案溯源**

- 每个答案返回引用来源列表。
- 来源至少包含 `doc_id`、`doc_name`、`chunk_id`、片段文本、相关性分数。
- PDF 如能提取页码，则展示页码；无法提取时不阻塞 MVP。
- 前端可展示答案正文和引用片段，用户能看到答案依据来自哪些文档。

**前端页面**

- 文档管理页：上传文档、查看文档列表、查看处理状态、删除文档。
- 对话页：输入问题、查看回答、查看引用来源。
- 基础配置页：展示当前默认模型、切片参数、检索 Top-K 等只读或简单可编辑配置。

**基础工程**

- 提供 Spring Boot 后端、React 前端和 Docker Compose 本地依赖环境。
- 本地环境至少能启动 MySQL、Redis、MinIO、Milvus。
- 提供 `.env.example` 或配置说明，避免真实密钥进入仓库。

### 2.2 本期不包含

- 完整用户体系、角色权限、租户隔离。
- 生产级安全审计、文件病毒扫描、敏感信息脱敏。
- 多模型高级路由、模型自动降级、成本优化策略。
- BM25、多路召回、概要索引和 BGE-Reranker 的完整实现。
- 复杂监控平台、链路追踪、告警系统。
- 生产级部署、灰度发布、自动扩缩容。
- 大规模文档集合优化，MVP 默认面向小型演示数据集。

## 3. 验收标准

### 3.1 文档处理

- 用户可以上传一个合法 PDF、Markdown 或 TXT 文件。
- 上传成功后，文档状态从 `PROCESSING` 变为 `READY`。
- 原始文件可以在 MinIO 中找到。
- MySQL 中存在对应文档记录和切片记录。
- Milvus 中存在与切片对应的向量记录。
- 文档处理失败时，状态变为 `ERROR`，并记录可排查的错误原因。

### 3.2 检索问答

- 用户可以在对话页提交问题并得到答案。
- 对已上传文档中的明确内容提问时，答案能基于相关片段生成。
- 每次问答请求会保存用户问题、助手回答、使用模型和引用来源。
- 当知识库没有相关内容时，系统不会编造答案，而是说明缺少足够依据。
- LLM 调用失败或超时时，前端能展示明确失败提示。

### 3.3 答案溯源

- 每个基于知识库生成的答案至少展示 1 个引用来源。
- 引用来源能展示文档名和原文片段。
- 点击或展开引用时，用户可以查看该片段的详细内容。
- 如果引用片段缺少页码，界面仍能正常展示文档名和片段内容。

### 3.4 配置与启动

- 新开发者按 README 或文档说明可以启动本地依赖服务。
- 后端启动后能连接 MySQL、Redis、MinIO、Milvus。
- 缺少 LLM API Key 时，后端启动应给出清晰配置错误，或相关功能返回明确提示。
- 配置文件不包含真实密钥。

### 3.5 基础前端交互

- 文档列表能显示文件名、格式、状态、上传时间和切片数量。
- 上传、删除、问答请求期间有加载状态。
- 失败场景有用户可理解的错误提示。
- 页面刷新后，已上传文档列表仍可从后端加载。

## 4. 异常处理

### 4.1 通用策略

- 用户可见错误信息保持简洁，说明发生了什么以及是否可以重试。
- 详细异常堆栈和外部服务响应写入后端日志，不直接暴露给前端。
- 可恢复失败应支持重试；不可恢复失败应标记状态并保留排查信息。
- 所有异步文档处理失败都必须更新 MySQL 状态，避免文档永久停留在 `PROCESSING`。

### 4.2 文档上传异常

| 场景 | 处理方式 |
| --- | --- |
| 文件为空 | 拒绝上传，返回“文件不能为空”。 |
| 格式不支持 | 拒绝上传，返回支持格式列表。 |
| 文件过大 | 拒绝上传，返回大小限制说明。 |
| MinIO 写入失败 | 文档不进入处理流程，返回上传失败并记录日志。 |
| 重复上传 | MVP 可允许重复上传；后续再加入 hash 去重策略。 |

### 4.3 文档解析与切片异常

| 场景 | 处理方式 |
| --- | --- |
| Tika 解析失败 | 文档状态置为 `ERROR`，记录 `error_msg`。 |
| 解析结果为空 | 文档状态置为 `ERROR`，提示未提取到可索引文本。 |
| 切片结果为空 | 文档状态置为 `ERROR`，记录切片失败原因。 |
| 单个切片过长 | 按默认最大 token 或字符数再次拆分。 |
| 页码或章节缺失 | 允许缺失，不阻塞索引构建。 |

### 4.4 向量化与索引异常

| 场景 | 处理方式 |
| --- | --- |
| Embedding 服务失败 | 文档状态置为 `ERROR`，允许后续重试处理。 |
| Milvus 不可用 | 文档状态置为 `ERROR`，记录连接或写入失败原因。 |
| 部分切片写入失败 | MVP 中整体标记失败，避免索引不完整导致结果不可解释。 |
| embedding 维度不匹配 | 阻止写入 Milvus，记录配置错误。 |

### 4.5 问答异常

| 场景 | 处理方式 |
| --- | --- |
| 用户问题为空 | 前端阻止提交，后端也做参数校验。 |
| 向量检索无结果 | 返回无足够依据的兜底回答。 |
| 召回分数过低 | 返回低置信兜底回答，并可展示最接近来源。 |
| LLM 超时 | 返回模型服务暂时不可用，记录超时日志。 |
| LLM 返回空答案 | 返回生成失败提示，保留检索来源用于排查。 |
| Redis 不可用 | 降级为无多轮记忆问答，不阻断单轮问答。 |

### 4.6 删除与一致性异常

删除文档时需要清理 MySQL、Milvus 和 MinIO 中的相关数据。MVP 默认执行顺序为：

1. 标记文档为 `DELETING`。
2. 删除 Milvus 向量。
3. 删除 MinIO 原始文件。
4. 删除或软删除 MySQL 文档与切片记录。

如果任一步失败：

- 文档状态标记为 `DELETE_FAILED`。
- 记录失败步骤和错误原因。
- 前端提示删除失败，可稍后重试。
- 不静默隐藏残留数据。

## 5. 默认实现取舍

- MVP 优先实现单模型、单知识库、单用户演示流程。
- 默认检索只使用向量召回，BM25 和 rerank 在后续阶段补充。
- 默认切片策略选择固定长度或滑动窗口，不强制实现语义切片和父子切片。
- 默认不做强权限控制，但所有接口预留后续接入鉴权的位置。
- 默认日志先使用应用日志，后续再接入指标系统和链路追踪。

## 6. 与主设计文档的关系

主设计文档描述完整目标架构，本文定义第一版落地边界。实施时以本文的 MVP 范围和验收标准为第一优先级；未进入 MVP 的能力应避免提前扩展，除非它们是完成主链路所必需的基础设施。
